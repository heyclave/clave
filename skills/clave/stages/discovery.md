# Stage: Discovery

**Reads:** the user's request (+ anything they share about the product/customers).
**Writes:** `docs/discovery.md`.

Capture the truth about the **product and its customers** before anything site-specific —
research, not a plan; true whether or not a website exists. The brief turns it into a site
plan.

## Discovery checklist

- [ ] **Draft from what's on disk first** — repo, `docs/assets/`, anything the user shared
      → never ask for something already available
- [ ] Ask only what the draft can't fill (**2–3 questions max**) — usually real customer
      pain or differentiation
      → phrase as confirmations of an inference ("audience is X — right?"), not blank
        prompts; corrections are higher-signal and less work
- [ ] **Ground in jobs-to-be-done** — what task is the customer hiring a solution for, and
      why; use the customer's own language where you have it
- [ ] Keep it lightweight — a one-pager, not a research report
- [ ] Write `docs/discovery.md` (template below)

## Write `docs/discovery.md`

```markdown
# Discovery — <Product Name>

## Product
<One or two sentences: what the product is and does.>

## Customer
<Who they are. The context/moment they're in.>

## Problem / job to be done
<The struggle they're trying to resolve. "When ___, they want to ___ so they can ___.">

## Differentiation
<Why this product over alternatives. What it does that others don't.>

## Proof points
<Evidence that backs the claims — outcomes, numbers, testimonials, credentials. Omit if none.>
```
